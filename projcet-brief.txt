📦 The Core Classes You'll Need
Start by thinking about what real e-commerce has, then simplify it:

Product — has a name, price, stock quantity, category
User — has a username, password, and a role (customer vs admin)
Cart — belongs to a user, holds items + quantities
Order — a "snapshot" of a completed cart with a status and timestamp
Store — the main engine that manages everything (products, users, orders)


🗺️ Suggested Build Order
Build it in phases so you always have something working:
Phase 1 — Products & Admin
→ Add, update, remove, and list products. Nothing fancy yet, just get OOP working.

Phase 2 — Users & Auth
→ Register, login, logout. Store passwords hashed (look up the hashlib or bcrypt module — great for your portfolio).
→ User Class attributes:
  -user_id
  -user_name
  -password
  -role

→ User Class methods:
  - check_password()
  - change_password()
  -is_Admin()

Note:
-> Methods shouldn't print() — that's the CLI's job.

Phase 3 — Shopping (Cart)
→ Logged-in users can browse, add to cart, remove from cart, see totals.

# constructor:
Cart is created → empty
    ↓
user calls add_item(product, quantity) → item added to dictionary
    ↓
user calls add_item(product, quantity) → another item added
    ↓
cart now has multiple items

→ Cart class methods:
  - add_item()
  - remove_item()
  - update_quantity()
  - calculate_total()
  - view_cart()
  - checkout():
       # 1. Copy the existing cart
       # 2. Loop through cart items and reduce stock for each product
       # 3. Clear the cart after
       # 4. Return something useful (i.e ordered_items)

Phase 4 — Orders
→ Cart becomes an Order. Users can view their order history.

Order.py
Attributes:
  - order_id
  - Who placed the order? (user_id)
  - What was ordered? (comes from checkout())
  - What's the total price?
  - When was it placed? (datetime.now())
  - What's the status? (e.g. "pending", "delivered")

For status — give it a default value in __init__ so the caller doesn't have to pass it every time. \

Phase 5 — Store.py / Storage
→ store.py is where everything comes together.
→ Think about it as the manager of the whole system. It holds all the data and coordinates between classes.

Store
├── manages → {products}  (dict of Product objects)
├── manages → {users}     (dict of User objects)
└── manages → {orders}    (dict of Order objects)


→ For attributes — what collections does a store need to keep track of?
  Think about what a real store manages:
    → Store name
    → All the products available
    → All registered users
    → All placed orders

→ Users dictionary ✅
    user_id as key, User object as value — clean and fast lookup.
→ Orders dictionary ✅
    order_id as key, Order object as value — same logic, perfect.
→ Products ✅
    A simple dictionary like users/orders would work fine here too:
    product_id as key, Product object as value.


→ For methods:

--> Product management methods:
   - add_product() first — the store receives product details and creates a Product object and adds it to self.products.
          # Arguments — the store should receive the raw details and create the Product itself.  
           - So arguments should be product_id, name, price, stock and inside the method you create Product        (product_id, name, price, stock).

           Duplicate check — what if a product with the same id already exists? 🤔

  - remove_product()

  - update_product() — an admin might want to update the name, price, or stock of an existing product.
        — what if the admin only wants to update the price but not the name or stock? 
        - You don't want to force them to pass all three values every time.
        - the clean approach is default parameters set to None:
            def update_product(self, id, name=None, price=None, stock=None):

--> User Mangement methods:
    - register_user() — creates a new user and adds to self.users
    - login_user() — finds the user and verifies their password

→ Customer can:
   → Browse/search products
   → Register/login
   → Manage their cart
   → Place orders
   → View their own orders

→ Save everything to JSON files or a SQLite database so data persists. (SQLite with Python's built-in sqlite3 module is a great portfolio flex.)
